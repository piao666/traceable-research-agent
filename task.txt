# Traceable Research Agent - Task Plan

更新时间：2026-06-29

## 项目定位

Traceable Research Agent 的核心目标是构建一个可追踪、可审计、可复盘的多工具调研 Agent。当前主线不是扩展第二套前端，也不是转向企业知识库 RAG 平台，而是继续强化：

- 任务规划与工具调用的稳定性
- Trace / Evidence / Report 的闭环
- MCP 只读工具协议互操作
- 真实演示问题下的端到端可靠性

Streamlit 继续作为主 demo UI；FastAPI 是稳定后端；MCP 端点用于外部客户端发现和只读调用。

## 当前已具备的闭环

- FastAPI 任务生命周期：create / plan / run / run_async / confirm / status
- Planned 与 ReAct 两种执行模式
- Tool Registry 统一注册本地工具和可选远端 MCP 工具
- file_reader / sql_query / rag_search / mcp_github_search / tavily_search / report_writer
- SQLGlot 只读校验、文件路径白名单、GitHub GET-only、HITL、ReAct 循环上限
- SQLite AgentRun / ToolTrace 持久化
- SSE 实时 Trace 事件流
- Streamlit 任务创建、计划查看、执行追踪、报告渲染
- EvidenceBundle 聚合、多来源证据分组、mock/fallback/failure 标记
- Evidence Export：JSON / JSONL / Markdown metadata、content preview、download
- MCP Server：只读工具发现、工具调用、trace_reader、report_reader
- MCP Client / Remote Tool Registry：可选发现安全远端只读工具

## 2026-07-01 - 板块三：MCP 通道升级

目标：把远端 MCP 从单一只读配置升级为通道化策略，保持默认安全、可审计、可接真实 HTTP JSON-RPC MCP 的工程子集，不做无限制 MCP hub。

规划实现：

- 新增远端 MCP 通道配置：
  - `MCP_CHANNEL_READONLY_SERVERS`
  - `MCP_CHANNEL_INTERACTIVE_SERVERS`
  - `MCP_CHANNEL_WRITE_SERVERS`
- 每个 server 支持 `name`、`base_url`、`transport`、`timeout_seconds`、`allowed_tools`、`blocked_tools`、`headers_env`、`channel`。
- 保留旧 `MCP_REMOTE_REGISTRY_ENABLED` / `MCP_REMOTE_SERVERS` 兼容入口，按 readonly channel 处理。
- `readonly` 通道只自动注册 `read_only=true`、`side_effect_free=true`、`requires_confirmation=false` 的远端工具。
- `interactive` 通道注册为需要 HITL，不进入默认只读 MCP 暴露，不进入 parallel 自动执行。
- `write` 通道默认禁用，即使配置也不自动注册可执行工具。
- 远端工具 trace metadata 写入 `tool_source=mcp_remote`、`remote_server`、`remote_channel`、`remote_tool_name`、`mcp_transport` 等字段。
- `/mcp/health` 返回 channel summary；`/mcp/tools` 显示 channel / policy / remote metadata。
- 新增 `scripts/smoke_mcp_channels.py` 覆盖 Firecrawl/browser/database/write fake server、allowlist/blocklist、headers_env 脱敏、HITL 和远端失败 trace。

安全边界：

- 不放松本地 `file_reader` path guard 和 SQL read-only guard。
- browser/playwright 类工具默认属于 interactive；write channel 默认不执行。
- `headers_env` 只读取环境变量值用于出站请求，不写入 trace/report/evidence。
- Streamlit 继续作为主 demo UI，不新增第二套前端。

## 2026-07-03 - 板块三补强：Deep Research Source Pack

目标：让 MCP 接入从“能接远端工具”升级为可解释的产品场景，优先提升搜索/调研深度，而不是把 browser/playwright 类高风险工具混入默认链路。

实现方向：

- 新增 Streamlit 场景模板：
  - 深度网页调研（Tavily + Firecrawl/Exa MCP）
  - 技术文档调研（GitHub + Context7/Exa MCP）
- 新模板使用 `allowed_tools=None`，由后端默认工具列表 + 已注册远端 MCP 工具动态决定可用工具；无远端 MCP 配置时仍可默认离线运行。
- Planner 识别 `deep_web_research` / `technical_docs_research`：
  - 深度网页调研优先 Tavily、Exa `web_search_exa/web_search_advanced_exa/web_fetch_exa`、Firecrawl `search/scrape/extract/map`。
  - 技术文档调研优先 GitHub、Context7 `resolve-library-id/query-docs`、Exa、Firecrawl。
  - 远端工具仍遵守 channel policy；interactive/write 不进入默认只读自动链路。
- EvidenceBundle 对 remote MCP 证据分层：
  - `mcp_remote_discovery`：search/map/resolve 类发现证据。
  - `mcp_remote_support`：scrape/extract/fetch/query-docs 类正文支撑证据。
  - `mcp_remote_failure`：失败远端调用，保留 trace，不升级为 API 500。
- Report 层识别 remote MCP URL，修复 LLM 输出中的工具名来源，例如 `来源：[firecrawl.scrape]`。
- `.env.example` / README 增加 Firecrawl、Exa、Context7 的 readonly 示例配置和 Firecrawl interact/browser 的 interactive 边界说明。

验证覆盖：

- `scripts/smoke_planner.py` 覆盖 fake Firecrawl/Exa/Context7 工具下的两个新模板。
- `scripts/smoke_mcp_channels.py` 覆盖 Firecrawl `search/scrape/map/extract/interact`、Exa、Context7、browser/database/write fake server。
- `scripts/smoke_evidence_aggregation.py` 覆盖 remote MCP discovery/support/failure 分类、URL 来源、headers_env 脱敏。
- `scripts/smoke_streamlit_frontend.py` 覆盖新模板、动态 allowed_tools、scenario_template_key。
