Reading conclusions:

- GPT Researcher is a web/local deep research agent. It focuses on turning a
  query into research questions, gathering sources, aggregating context, and
  producing cited reports.
- The README architecture uses planner + execution agents + publisher/report
  writer. This is useful for our planner/executor/reporter mental model.
- `main.py` is thin: load `.env`, import `backend.server.app`, and start uvicorn
  on `0.0.0.0:8000`.
- `backend/server/app.py` uses FastAPI, CORS, static files, report endpoints,
  `/report/`, `/api/reports`, `/api/chat`, `/files/`, `/upload/`,
  file deletion, and WebSocket communication. It uses a lightweight
  `reports.json` style report store rather than a task/trace database.
- `gpt_researcher/agent.py` defines `GPTResearcher`, initializes config,
  retrievers, memory, ResearchConductor, ReportGenerator, ContextManager,
  BrowserManager, SourceCurator, optional image generation, and MCP strategy.
  `conduct_research()` handles agent selection and context gathering.
  `write_report()` delegates final report generation. `research_costs` and
  `step_costs` track LLM cost, but not structured per-tool execution traces.
- `gpt_researcher/skills/researcher.py` branches by source: web, local,
  hybrid, Azure, LangChain documents, and vector store. It plans sub-queries,
  searches/scrapes, uses vector store when configured, and integrates MCP
  retrievers with fast/deep/disabled style behavior.
- `gpt_researcher/skills/writer.py` builds report parameters from query, role,
  report type/source, tone, websocket, config, and headers. It generates final
  report, introduction, conclusion, subtopics, and draft section titles.
- `gpt_researcher/document/document.py` walks local paths or file lists and
  dispatches loaders by extension. Supported formats include PDF, txt, doc,
  docx, pptx, csv, xls/xlsx, md, html, and htm.
- `gpt_researcher/mcp/*` manages MCP server config conversion, client
  lifecycle, LLM-based tool selection with fallback pattern matching, tool
  execution, result normalization, WebSocket streaming, and security guidance.
- Docker runs backend on 8000 and Next.js frontend on 3000, mounting local
  `my-docs`, `outputs`, and `logs`, with API keys supplied through environment
  variables.

What can be borrowed:

- Planner/executor/reporter conceptual split.
- Multi-source research pattern: web/local/hybrid/vector/MCP.
- Report generation from task, role, context, and evidence rather than raw
  output concatenation.
- Local document format support ideas.
- MCP config concepts, client lifecycle, read-only token handling, and fallback
  tool selection ideas.
- Docker pattern of mounting input/output/log directories and injecting secrets
  through environment variables.

What must be redesigned:

- Build an independent FastAPI API around runs, statuses, tool registry, traces,
  and reports instead of reusing `/report/` and `reports.json`.
- Persist every tool call as a `tool_traces` row; GPT Researcher logs and cost
  counters are not enough.
- Represent file, SQL, RAG, GitHub/MCP, and report writing as registered tools
  with structured input/output schemas.
- Add explicit safety gates for file paths, read-only SQL, high-risk tools, and
  later human confirmation.
- Start with deterministic JSON plans before autonomous LLM tool selection.
- Keep MCP/GitHub as a later read-only enhancement so Phase 1-3 are not blocked.

## Project Planning Document Reading Notes

Status: read on 2026-06-15 using Python `python-docx` from the bundled Codex
document runtime. The file was located with a `Traceable_Research_Agent_*.docx`
glob to avoid Chinese filename encoding issues.

Project positioning:

- This repository is the second independent Agent engineering project.
- It should not be implemented inside another Agent+RAG backend repository.
- It references GPT Researcher only for design ideas.
- The core value is task execution, tool safety, trace persistence, backend
  APIs, and evaluation.

Final target:

- Accept complex research tasks and create a `run_id`.
- Maintain pending/running/completed/failed/waiting_human state.
- Plan steps, execute registered tools, collect evidence, and generate
  Markdown reports.
- Expose task creation, status query, trace query, report query, and tool-list
  APIs.
- Provide smoke demos, eval cases, README, trace examples, bad cases, and
  interview-ready explanations.

API design:

- `GET /health`
- `POST /api/tasks`
- `GET /api/tasks/{run_id}`
- `GET /api/tasks/{run_id}/trace`
- `GET /api/reports/{run_id}`
- `GET /api/tools`
- `POST /api/tasks/{run_id}/confirm`

DB design:

- `agent_runs` stores run-level state, task, plan, allowed tools, progress,
  report path, errors, total tool calls, latency, estimated cost, and timestamps.
- `tool_traces` stores one row per tool call with input/output summaries, raw
  JSON, status, latency, errors, token/cost estimates, and timestamps.
- Optional later table: `prompt_versions`.

Tool system:

- Tool Registry with `ToolSpec`, `ToolResult`, `register_tool`, `list_tools`,
  and `execute_tool`.
- Minimum tools: `file_reader`, `sql_query`, `rag_search`, `report_writer`.
- Later enhancement: read-only `mcp_github_search` via GitHub API/mock first;
  MCP integration is optional and should not block MVP.

Trace requirements:

- Every tool call must be persisted.
- Failed calls must also generate traces.
- Safety rejections must be visible through traces.
- Trace API is a core project feature, not just logging.

Evaluation requirements:

- At least 10 eval cases: file reading, SQL, RAG, mixed tasks, and safety/failure
  tasks.
- Minimum target: 7+ successful tasks, all failure cases visible, and all tool
  calls trace-complete.
- Preserve 2-3 bad cases for README and interview explanation.

## Global Constraints

- Standalone repo, not `agent-service-toolkit`.
- No company-specific names in code, docs, commit messages, examples, or demo
  data.
- No secrets, tokens, `.env`, private data, caches, local DB files, bulky
  generated reports, or model artifacts committed.
- Tools are read-only by default.
- File tools must enforce `workspace/docs` whitelist and path traversal
  protection.
- SQL tools must allow only `SELECT`/`WITH` and reject destructive statements.
- MCP/GitHub tools must be read-only and must not block MVP.
- Every tool call must create a trace record.
- Every phase must end with a checkpoint commit and GitHub push.

## 2026-06-17 - Day20+ Revised Master Roadmap: LLM Planner, Streamlit UI, Real RAG, And Engineering Hardening

This revised roadmap supersedes previous Day20+ priority ordering.

### 1. Overall Principles

Current Day19 version is already a complete demo-ready MVP and interview
project. Later optimization should not replace the existing architecture.
Instead, it should enhance the current main chain:

```text
Planner -> Executor -> Tool Registry -> Trace DB -> Reporter
```

Future work is reorganized into three major steps:

1. Step 1: LLM Planner Upgrade with DeepSeek/Qwen and deterministic fallback.
2. Step 2: Streamlit Frontend Demo Layer.
3. Step 3: Real RAG Upgrade plus Engineering Hardening.

Priority:

1. Build the LLM Planner first because it upgrades the core Agent planning
   capability.
2. Build the Streamlit frontend second because it wraps the existing API flow
   into an interview-friendly demo UI.
3. Build real RAG and engineering hardening last because they involve heavier
   model dependencies, vector database changes, migrations, async execution,
   and auth.

All later work must keep these boundaries:

- Do not break the existing FastAPI API.
- Do not change the core boundary of `POST /api/tasks`: it only creates a
  pending run and persisted plan; it does not auto-run tools.
- Tool execution still goes through `POST /api/tasks/{run_id}/run` or explicit
  tool execute endpoints.
- All tool success, failed, and rejected results must continue to be written to
  `tool_traces`.
- The deterministic planner must always remain available as fallback.
- The deterministic/json RAG path must always remain available as fallback.
- Streamlit must call APIs only; it must not directly access the database or
  internal functions.
- Without API keys, local models, or extra dependencies, the project must still
  run in Day19 lightweight mode.
- Docker default mode must still start in lightweight mode.
- All old smoke/eval checks must continue to pass.
- Model files, API keys, runtime DBs, RAG indexes, Chroma/FAISS indexes,
  reports, and eval outputs must not be committed to Git.

### 2. Step 1: LLM Planner Upgrade

Goal:

Upgrade the current deterministic rule-based Planner to an optional LLM Planner
that supports DeepSeek and Qwen while preserving deterministic fallback.

Core value:

- Planning no longer depends only on keyword rules.
- Users can describe tasks more naturally and still get multi-tool plans.
- The project gains a real LLM Agent Planner talking point.
- Deterministic fallback preserves stability and demo reliability.

#### 2.1 LLM provider abstraction

#### 2.2 Strict plan JSON schema

- `notes`
- `confirmation`
- `planner_source`

Step schema:

- `goal`
- `tool_name`
- `arguments`
- `expected_output`
- `completion_criteria`
- `risk_level`
- `requires_confirmation`

Validation requirements:

- Parse LLM output as JSON.
- Validate required fields.
- Validate against the tool whitelist.
- Fallback to deterministic planner when output is invalid.

#### 2.3 Planner integration

- Add `planner_mode` to `plan_task`:
  - `deterministic`
  - `llm`
  - `auto`
- Default mode remains deterministic.
- Auto mode tries LLM only when `LLM_PLANNER_ENABLED=true` and a key is
  available; otherwise it uses deterministic fallback.
- LLM output must be filtered by `allowed_tools`.
- Unknown tools must be filtered or recorded in notes.

二、Day32-A：Optional ReAct Executor With Thought/Action/Observation Trace

目标：
新增可选 ReAct executor，不替换现有 planned executor。

核心设计：

* 增加 EXECUTION_MODE=planned|react。
* planned 继续作为默认稳定模式。
* react 模式中，每一步由 LLM 根据 task、available tools、observation_history 决策下一步。
* 每一步输出结构化 JSON：

  * thought
  * action
  * args
  * finish_reason
* action 可以是 tool name 或 finish。
* action 必须经过 allowed_tools 校验。
* args 必须经过 tool schema 或工具自身校验。
* 每个 tool result 写入 observation_history。
* 每个 ReAct step 写入 trace。
* 支持 max_steps，例如 8。
* 支持 same_tool_max_calls，例如 3。
* 工具失败、RAG no hits、SQL rejected、GitHub fallback 等 observation 要能反馈给 LLM，用于下一步重新决策。
* HITL 节点不能被绕过。
* SQL parser、file path safety、GitHub read-only policy 等安全边界继续生效。
* 如果 ReAct LLM 输出非法 JSON 或未知工具，应稳定 fallback 或 finish with limitation，不能导致 API 500。
* Streamlit 后续需要能展示 Thought / Action / Observation trace。

预期新增：

* app/agent/react_executor.py 或等价模块
* ReAct prompt 模板
* ReAct decision schema
* ReAct trace metadata
* scripts/smoke_react_executor.py
* eval cases for react mode
* README/task 更新

验收标准：

* planned 模式旧流程全部通过。
* react 模式能完成至少一个完整 research task。
* trace 中能看到 thought/action/observation。
* 工具失败后 react 模式能换工具、提前结束或给出合理 limitation。
* repeated / max_steps / same_tool_max_calls 有保护。
* HITL 不被绕过。
* 旧 smoke/eval 全部通过。
