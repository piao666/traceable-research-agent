### 核心代码 Bug 修复（均已通过实际运行验证）

#### BUG-1：create_llm_client() 缺少必填参数 settings
影响范围：所有任务执行后报告生成阶段
根因：executor.py 第 166 行调用 create_llm_client()，
      react_executor.py 第 229 行同样问题。
      create_llm_client(settings) 的 settings 是必填参数，
      缺少时抛 TypeError，被外层 except Exception 捕获，
      任务状态直接变为 failed，报告永远不生成。
验证：python3 -c "create_llm_client()" 实际运行确认报错。
修复：executor.py 改为 create_llm_client(_exec_settings)
      react_executor.py 改为 create_llm_client(settings_obj)
影响后果：修复前所有任务必然失败，这是 executor.py 上线后的致命 Bug。

#### BUG-2：execution_mode_override 参数接收但未写入 plan
影响范围：Streamlit 执行模式切换完全无效
根因：planner.py 的 plan_task() 接收 execution_mode_override 参数，
      但 4 条 return 路径均直接 return plan，
      从未将 override 写入 plan["execution_mode"]。
验证：plan_task(..., execution_mode_override='react') 后
      读 plan["execution_mode"] 得到 None，实际运行确认。
修复：添加 _apply_execution_mode(plan, override) helper，
      在 4 个 return 路径前调用：
      - 确定性早返回
      - LLM 成功返回
      - LLM fallback 返回
      - 末端确定性返回

#### BUG-3：_render_final_answer 函数体被截断，隐式返回 None
影响范围：所有报告生成崩溃
根因：某次 Phase A 补丁将 _SYNTHESIS_SYSTEM 常量和
      _llm_synthesize_answer 函数插入到 _render_final_answer
      函数体内部，导致函数在第 3 行后"自然结束"，
      剩余 finish_record / github_record 等处理逻辑
      变成孤立的模块级代码（语法合法但不属于任何函数）。
      函数隐式返回 None，list literal 中的 *None 崩溃。
验证：inspect.getsource(_render_final_answer) 只返回 3 行。
      _render_final_answer('test', [], []) 实际返回 None。
修复：重建整个 reporter.py 结构：
      Phase A 代码（_SYNTHESIS_SYSTEM、_llm_synthesize_answer）
      放在模块级正确位置；
      _render_final_answer 完整恢复所有业务逻辑；
      *_render_final_answer(...) 从 list literal 提取为变量，
      加 or [] 防 None。

#### BUG-4：Phase A LLM 合成未真正集成进 generate_markdown_report
影响范围：LLM 合成报告功能未生效
根因：Phase A 补丁针对 final_answer_lines = ... 赋值模式注入，
      但实际代码使用的是 *_render_final_answer(...) 内联模式，
      模式不匹配导致注入静默失败。
修复：重写 generate_markdown_report 末尾逻辑：
      先尝试 _llm_synthesize_answer，结果赋给 _llm_answer，
      再决定 _final_answer_lines 的内容，
      最后展开到 lines 列表。

#### BUG-5：reporter.py 缺少 context_compressor import
影响范围：_llm_synthesize_answer 调用时 NameError
根因：compress_evidence / has_useful_evidence 被调用，
      但 from app.agent.context_compressor import ... 行缺失。
验证：run_plan 运行后 error_message = "name 'has_useful_evidence' is not defined"
修复：在 reporter.py 第 18 行后添加正确 import。

#### BUG-6：dispatcher.py 读 settings 不读 plan 执行模式
影响范围：UI 模式切换无效（与 BUG-2 协同）
根因：dispatcher.py 第 18 行只读 settings_obj.execution_mode（来自 .env），
      即使 plan["execution_mode"] 已经写入，也被完全忽略。
修复：dispatcher 优先读 plan["execution_mode"]，
      有值时用 plan 的模式，否则才 fallback 到 settings。

#### BUG-7：planner_client.py mcp_github_search mode 硬编码为 mock
影响范围：即使 source_mode=real 也走 mock，GitHub 搜索无实际 API 调用
根因：tool_defaults["mcp_github_search"]["mode"] = "mock" 写死。
修复：改为动态：
      "mode": "public_api" if source_mode == "real" else "mock"

### Phase C：并行工具执行（预计 2-3 天）

目标：把串行 for 循环改为 asyncio.gather 并发执行，
      速度提升 3-5x（对有网络延迟的工具效果最显著）。

实现方案：
新增文件：app/agent/parallel_executor.py

```python
async def run_steps_parallel(steps, db, run_id, max_concurrent=3):
    semaphore = asyncio.Semaphore(max_concurrent)
    async def _run_one(step):
        async with semaphore:
            loop = asyncio.get_event_loop()
            result = await loop.run_in_executor(
                None, execute_tool, step["tool_name"], step["arguments"]
            )
            record_trace_event(db, run_id, step["step_no"], ...)
            return {"step": step, "result": result}
    return await asyncio.gather(*[_run_one(s) for s in steps])
```

并行策略：
- rag_search + tavily_search + mcp_github_search 并行
- sql_query 串行（同一 SQLite 连接）
- report_writer 必须等其他工具完成后执行

### Phase D：WebSocket 实时进度推送（预计 2 天）

目标：替换 REST 轮询（每 3s 一次），改为 WebSocket 实时推送每步进度，
      面试演示时可以实时看 Agent 思考过程。

实现方案：
新增文件：app/api/ws.py

```python
@router.websocket("/ws/{run_id}")
async def task_websocket(websocket: WebSocket, run_id: str):
    await websocket.accept()
    active_connections[run_id] = websocket
    while True:
        await asyncio.sleep(0.5)
        traces = get_latest_traces(run_id)
        status = get_run_status(run_id)
        await websocket.send_json({
            "type": "trace_update",
            "traces": traces,
            "status": status,
        })
        if status in ("completed", "failed"):
            break
```

Streamlit 侧：每 500ms 通过 websocket 接收事件，实时渲染卡片。

## 2026-06-27 - Differentiation Roadmap: Parallel Tools And Full MCP Service

### 背景与定位调整

参考另一个项目 `agent-service-toolkit/feature/enterprise-rag-backend` 后，当前项目需要避免继续和企业知识库 RAG 后端高度重叠。

`agent-service-toolkit` 更适合承担：
- 企业知识库 RAG 后端
- 文档解析、结构化检索、metadata / symbol retrieval
- 多轮记忆、custom graph、evidence verifier
- 大规模企业知识库 eval 和 bad-case 诊断
- 面向企业 KB 问答的稳定服务接口

`Traceable Research Agent` 后续应强化差异化方向：
- 多工具并行 Research Agent
- Planned / ReAct 双执行模式
- 完整 Trace、HITL、安全边界和可审计报告
- MCP-first 工具协议化
- 完整 MCP Server / MCP Client / remote tool registry
- GitHub / Tavily / RAG / SQL / file 等多源工具并发编排
- 实时执行进度展示
- Streamlit 继续作为主展示前端，轻量 Web UI 仅作为后期扩展

核心定位调整为：

> 一个可追踪、多工具并行、MCP-first 的 Research Agent 执行系统，而不是另一个企业知识库 RAG QA 后端。

### Phase C：Parallel Multi-tool Execution

目标：
- 将串行工具执行升级为可控并行执行。
- 优先提升 GitHub / Tavily / RAG 等网络或检索工具的整体耗时。
- 保持 planned executor 的稳定性和 Trace 完整性。

建议新增：
- `app/agent/parallel_executor.py`

配置建议：
```ini
PARALLEL_EXECUTION_ENABLED=false
PARALLEL_MAX_CONCURRENT=3
PARALLEL_TOOL_TIMEOUT_SECONDS=60
```

并行策略：
- 可并行：
  - `rag_search`
  - `tavily_search`
  - `mcp_github_search`
  - `file_reader`
- 保持串行或 barrier：
  - `sql_query`：SQLite / SQL 安全边界下保持串行更稳。
  - `report_writer`：必须等待所有 evidence tools 完成。
  - HITL confirmation：不得绕过人工确认。

Trace 要求：
- 记录 parallel_batch_id。
- 记录 tool start/end time。
- 记录 latency、status、metadata。
- 记录 dependency / barrier 信息。

验收标准：
- 一个任务中 GitHub + Tavily + RAG 可以并行执行。
- 并行执行耗时低于串行基线。
- Trace 不丢失、不乱写。
- report_writer 等待全部证据完成后再生成报告。
- `/run` 和 `/run_async` 接口保持兼容。

### Phase D：Full MCP Server Foundation

目标：
- 从当前 MCP-style read-only adapter 升级为完整 MCP Server。
- 将项目内工具协议化暴露给外部 MCP client。
- 默认 read-only，但 read-only 不等于只能 GET；核心含义是“不产生外部状态变更”。

建议新增：
- `app/mcp/server.py`
- `app/mcp/policy.py`
- `app/mcp/schemas.py`

MCP tools 初始暴露：
- `file_reader`
- `rag_search`
- `sql_query_readonly`
- `github_search`
- `tavily_search`
- `trace_reader`
- `report_reader`

MCP tool metadata 必须包含：
- name
- description
- input_schema
- output_schema
- read_only
- side_effect_free
- requires_confirmation
- risk_level

Read-only policy 说明：
- GitHub 搜索可以继续使用 GET-only。
- Tavily Search 语义上是只读搜索，即使 HTTP method 是 POST，也应允许。
- 禁止所有会改变外部状态的操作：
  - GitHub issue create
  - PR comment
  - repository mutation
  - push
  - DB write
  - 文件写入非允许路径

验收标准：
- 外部 MCP client 可以发现 tools。
- 可以通过 MCP 调用只读工具。
- MCP 调用结果写入 Trace。
- 没有任何 write-capable tool 被默认暴露。
- Tavily 搜索不因 read-only policy 被误杀。

### Phase E：MCP Client And Remote Tool Registry

目标：
- 当前 Agent 不只执行本地工具，也能调用远端 MCP tools。
- 本地工具和远端 MCP 工具在 Planner / ReAct 看来是统一 Tool Registry。

建议新增：
- local tool registry
- mcp remote registry
- merged tool registry

行为要求：
- Planner 能看到 local + remote MCP tools 的统一描述。
- ReAct action 仍必须通过 allowed_tools 校验。
- 远端 MCP tool timeout / failure 作为 Observation 写入 ReAct history。
- Trace 中区分：
  - `tool_source=local`
  - `tool_source=mcp_remote`

验收标准：

## 2026-06-28 - Day37 Parallel Multi-tool Execution

### Why parallel execution was added

Traceable Research Agent is moving toward a traceable, multi-tool Research Agent runtime.
Independent evidence tools such as GitHub search, Tavily search, RAG retrieval, file reads,
and read-only SQL should not always pay serial latency. Day37 adds an optional planned
parallel executor while preserving Trace, HITL, read-only safety, and report generation.

### Default serial, optional parallel

- Default remains `PARALLEL_EXECUTION_ENABLED=false`; the existing serial planned executor
  is still the stable path.
- When `PARALLEL_EXECUTION_ENABLED=true` and the plan uses `execution_mode=planned`,
  dispatcher calls `run_plan_parallel`.
- ReAct remains a dynamic serial loop in Day37.

### Parallel tool scope

Eligible independent read-only tools:
- `file_reader`
- `rag_search`
- `mcp_github_search`
- `tavily_search`
- read-only `sql_query`

Never parallelized:
- `report_writer`, which remains a final barrier after evidence collection.
- `requires_confirmation=true` HITL steps, which must stop at `waiting_human`.
- Steps with explicit dependency keys such as `depends_on` or `after_step`.

### Trace metadata

Parallel execution metadata is stored in existing trace output metadata without changing
the database schema:
- `parallel=true`
- `parallel_group_id`
- `parallel_worker_id`
- `parallel_group_size`
- `execution_mode=planned_parallel`
- `started_at`
- `finished_at`
- `latency_ms`

Reporter and Streamlit display the parallel group, worker, and latency information.
Failed tools remain visible as failed traces and in the report; they do not turn the
whole API request into a 500.

### smoke_parallel_execution.py result

`scripts/smoke_parallel_execution.py` covers:
- default serial guard
- parallel group creation
- trace metadata
- report_writer barrier
- failed tool visibility
- HITL guard
- `/run_async` parallel dispatch

Expected output:

```json
  "parallel_group": "ok",
  "trace_metadata": "ok",
  "report_writer_guard": "ok",
  "failure_visible": "ok",
  "hitl_guard": "ok",
  "async_dispatch": "ok"
}
```

Default eval remains offline-friendly and does not require real external network,
