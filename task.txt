# Traceable Research Agent：闭环验证记录与 P0-P5 实施规划

更新时间：2026-07-16

## 1. 当前项目结论

项目已经形成可运行的 Demo 闭环：创建任务 -> 生成计划 -> 执行工具 -> 持久化 Trace -> 聚合证据 -> 生成并导出报告，同时具备异步任务、SSE 状态流、HITL 确认、MCP 工具桥接、SQL/RAG/GitHub 等工具接入和基础评测能力。

当前闭环解决的是“任务可以完成、过程可以观察、结果可以导出”的工程问题，但还没有完成 Deep Research 所需的三个核心能力：Claim 级来源可靠性计算、观点冲突检测与调和、报告结论到不可变原文片段的完整引用链。

### 2.3 当前闭环的已知边界

- `EvidenceBundle` 当前主要按 Plan step 关联证据，并非报告级原子 Claim 图谱。
- 证据置信度仍以成功、mock、fallback 等状态做粗粒度判断，没有基于来源和内容的校准评分。
- 尚未检测 RAG、SQL、网页和 GitHub 数据之间的语义冲突。
- Reporter 消费压缩后的文本上下文，不能保证每个报告事实都绑定可验证引用。
- 网页引用主要依赖 URL，网页变化后无法证明生成报告时实际看到的内容。
- Tool Registry 仍是原子工具层，没有任务级 Skill 的输入、输出、策略、预算和质量门禁。
- 当前 ReAct/Planned 评测使用 deterministic fake decision，尚不能证明多工具方案在真实研究质量上优于单 Tavily。
- SQLite 和 FastAPI BackgroundTasks 适合 Demo，不满足长期审计、水平扩展和崩溃恢复要求。

## 4. P1：建立 Claim 级证据与引用链

目标：将当前以 ToolTrace/Plan step 为中心的证据组织升级为以原子 Claim 为中心、能够反向审计的证据图谱。

### 4.1 目标数据链

`ToolTrace -> SourceDocument -> SourceSnapshot -> Passage -> EvidenceAssertion -> ResearchClaim -> ClaimEvidenceEdge -> Citation -> ReportClaim`

### 4.2 实施任务

- 新增 `SourceDocument`：规范化来源身份、URI、Provider、来源类型和所属组织。
- 新增不可变 `SourceSnapshot`：抓取时间、内容哈希、提取器版本、原始制品路径、Trace ID 和提取状态。
- 新增 `Passage`：保存证据片段、片段哈希及可重放定位器。
- 新增 `EvidenceAssertion`：主体、谓词、值、单位、时间、限定条件、极性和提取置信度。
- 新增 `ResearchClaim` 和 `ReportClaim`，区分研究阶段事实与最终报告实际陈述。
- 新增 `ClaimEvidenceEdge`，表达 `supports`、`refutes` 和 `contextualizes` 关系。
- 新增 `Citation`，将报告 Claim 绑定到 Passage、Snapshot 和 ToolTrace。
- 为 Web、RAG、SQL、GitHub 分别实现 ToolResult 标准化适配器。
- Web 定位器保存 URL、快照、字符区间；RAG 保存 corpus/document/chunk/offset；SQL 保存查询哈希、数据快照、表列和行标识；GitHub 保存 repo/commit/path/line。
- SQLite 只保存索引和结构化字段，大正文按 SHA-256 压缩存入 `workspace/artifacts/`。
- 增加数据库 schema 版本和迁移脚本，保留旧 `EvidenceItem` API 的 V1/V2 兼容适配器。
- Reporter 只接受结构化证据和有效 Citation ID，报告完成后执行引用完整性校验。

### 4.3 P1 复验

- 对四类来源适配器执行正常、空结果、缺字段、超长正文和非法定位器单元测试。
- 验证每条 EvidenceAssertion 都能反查 Passage、Snapshot 和 ToolTrace。
- 验证每条 Citation 都能定位到当时使用的不可变原文，而不是只返回当前 URL。
- 注入 LLM 编造的 Citation ID，确认报告校验失败或降级，不能静默接受。
- 使用 Web、RAG、SQL、GitHub 混合任务跑 E2E，逐条反查报告关键事实。
- 使用 20、100、1000 条工具证据检查批量写入、索引查询和 N+1 问题。
- 重跑旧证据 API、导出、前端和报告下载，确保兼容。

### 4.4 P1 完成标准

- 100% EvidenceAssertion 关联原始 Passage 和 ToolTrace。
- 100% 报告 Citation 可反向解析，不存在悬空引用。
- 原始正文不会导致 SQLite 体积无控制增长。

## 5. P2：来源可靠性、去重与冲突调和

目标：使系统能够解释“为什么相信某条证据”，识别多源矛盾，并在无法裁决时明确保留不确定性。

### 5.1 实施任务

- 增加版本化 `source_policy` 配置，按 Claim 类型配置来源先验、时效要求、允许域、禁用域和阈值。
- 初始可靠性公式：权威性 25%、可溯源性 20%、时效性 15%、Claim 相关性 20%、来源独立性 10%、提取完整性 10%。
- 根据 Claim 类型调整来源优先级：财务数据优先监管披露和受治理 SQL，产品能力优先官方文档，社会评价不能因来自官方渠道直接获得最高客观性。
- 实现 URL 规范化、内容哈希、转载识别、同组织聚类和独立来源簇。
- 规范化实体别名、币种、单位、百分比、时间范围、版本、正负极性和统计口径。
- 优先使用确定性数值、时间、范围和极性规则识别冲突。
- LLM 仅作为候选语义关系分类器，必须返回结构化关系和理由，不直接充当最终事实裁判。
- 实现冲突状态机：`no_conflict`、`resolved_by_scope`、`resolved_by_authority`、`resolved_by_recency`、`unresolved`、`requires_human`。
- 按独立来源簇聚合支持和反驳质量，不对冲突数值简单求平均。
- Reporter 必须显式展示未解决冲突、适用口径和可信度限制。
- 每次评分保存分项结果、策略版本、计算时间和调和原因，确保可解释和可复算。

### 5.2 P2 复验

- 建立数值相反、时间不同、单位不同、转载重复、官方更正、SQL/网页冲突、无法裁决七类黄金样例。
- 确定性样例的冲突识别和状态转换必须 100% 符合预期。
- 同一文章转载十次时，独立来源数量仍应为一。
- 修改策略权重后不改核心代码即可改变评分，并保留策略版本。
- 相同输入重复运行时，非 LLM 评分和冲突结果必须一致。
- LLM 输出非法 Schema 时必须降级为 `unresolved`，不能自动选边。
- 高置信结论必须满足最低独立来源数和最低可靠性阈值。
- 以 10,000 条 Evidence 做索引查询压测，本地 Claim 证据查询目标 `p95 < 300ms`，不包含外部 LLM 时间。

### 5.3 P2 完成标准

- 系统能区分真实矛盾、口径差异和补充上下文。
- 所有可靠性评分可解释、可复算、可通过配置调整。
- 未解决冲突不会被报告隐藏或表述成确定事实。

## 6. P3：从原子工具升级为任务级 Skills

目标：将“模型自由拼工具”升级为可复用、可治理、可评测的研究任务能力。

### 6.1 实施任务

- 定义 `ResearchSkillSpec`：名称、版本、输入/输出 Schema、允许工具、来源策略、最少独立来源数、DAG、重试、fallback、预算、超时、确认规则、质量门禁和评测 Profile。
- 首批实现 `web_fact_check`、`company_deep_research`、`metric_reconciliation` 三个 Skill。
- Planner 优先选择 Skill，只有 Skill 内部才编排原子 Tool。
- 将 `allowed_tools`、`risk_level`、`requires_confirmation`、内容访问策略统一放入 Policy Engine，在计划解析和执行层双重强制。
- 支持“禁止查询包含薪资关键词文档”等配置化规则，并记录策略拒绝 Trace。
- 增加 Skill 预算控制、幂等键、重复动作检测、re-planning 和 fallback 链。

### 6.2 P3 复验

- 验证三个 Skill 的输入/输出契约、成功路径、超时、fallback 和预算耗尽。
- 验证 Planner 不能绕过 Skill 调用未授权工具。
- 修改内容策略配置后无需修改核心代码即可生效。
- 对重复工具调用、相同 Observation 和循环计划执行死循环防护测试。
- 比较原子工具模式与 Skill 模式的步骤数、失败恢复率和 Trace 可读性。

### 6.3 P3 完成标准

- 常见研究任务可以通过版本化 Skill 稳定复用。
- 工具权限和内容策略无法被 Prompt 或 Planner 绕过。
- Skill 的每次执行均有预算、质量门禁和完整 Trace。

## 7. P4：建立标准 Deep Research 评测体系

目标：用可复现实验回答“多工具、MCP、证据评分和冲突处理是否真的优于单 Tavily”。

### 7.1 实施任务

- 建立覆盖事实核查、公司研究、指标对账、技术调研和冲突信息的标注数据集。
- 设置四组消融实验：A 单 Tavily；B 多原子工具；C 多工具 + MCP；D 完整可靠性/冲突/引用链流水线。
- 指标包括：事实正确率、引用准确率、引用覆盖率、权威来源占比、独立来源覆盖、无证据 Claim 率、冲突 Precision/Recall、冲突调和准确率、置信度校准 ECE/Brier、任务成功率、延迟、成本和恢复率。
- 建立 deterministic 回归集、真实 Provider 固定快照集和人工盲评集。
- 保存数据集版本、Prompt、模型、Provider、策略、Skill 和代码 Commit，确保实验可复现。
- 输出机器可读结果、对比表、失败案例和置信区间，不只输出单个平均分。

### 7.2 P4 复验

- 同一固定快照重复评测，核心指标波动应在设定范围内。
- 随机抽查引用，验证引用片段确实蕴含对应 Claim。
- 对评分权重、来源去重、冲突检测和 Skill 编排分别做消融。
- 由人工标注者复核冲突和引用错误，计算一致性。
- 初始质量目标：关键 Claim 引用覆盖率 >= 95%，引用蕴含准确率 >= 90%，无证据事实 <= 3%，冲突召回率 >= 85%；最终阈值以标注集校准结果为准。

### 7.3 P4 完成标准

- 能用数据说明每项能力带来的质量、延迟和成本变化。
- 评测结果可在固定版本环境中复现。
- 发布前质量门禁由指标驱动，而不是依赖 Demo 主观观察。

## 8. P5：生产化存储、调度与治理

目标：支持六个月以上审计保留、任务恢复、多实例部署和租户隔离。

### 8.1 实施任务

- 将任务、Trace、Claim、Citation 和策略审计数据迁移到 PostgreSQL。
- 将网页快照、报告、导出和大体积 ToolResult 放入 S3/MinIO 等对象存储。
- 使用 Redis/RabbitMQ 与 Celery/Dramatiq/Arq 等持久任务队列替代 FastAPI BackgroundTasks。
- 实现 worker lease、heartbeat、幂等执行、断点续跑、死信队列和任务取消。
- 为 SSE 增加断线重连、Last-Event-ID 和长期任务事件保留。
- 实现租户级 RBAC、行级访问控制、MCP Client 身份认证和审计查询授权。
- 实现 Trace 生命周期管理：热数据、冷归档、保留期、合法冻结、自动清理和删除审计。
- 增加 OpenTelemetry、指标、结构化日志、告警、备份恢复和容量监控。
- 完善 Docker Compose/生产部署配置和数据库迁移流程。

### 8.2 P5 复验

- 在工具执行、证据写入和报告生成阶段分别杀死 worker，验证任务能够恢复且不重复产生副作用。
- 重启 API、队列、数据库和宿主机，验证任务与 Trace 不丢失。
- 执行多租户越权测试，恶意 MCP Client 不能读取其他租户 Trace。
- 模拟六个月数据量，测试分区、归档、查询、清理和恢复时间。
- 执行 PostgreSQL 备份恢复演练和对象存储快照恢复。
- 执行并发、限流、队列堆积、OOM、Provider 故障和网络分区压测。

### 8.3 P5 完成标准

- 任务在单个容器或宿主机故障后可以自动恢复。
- 六个月审计数据可查询、可归档、可按策略清理。
- 租户隔离、授权、备份恢复和可观测性达到生产部署要求。

## 9. 推荐实施顺序与依赖

1. P0 必须先完成，否则后续实验会被隐式模型调用和配置漂移污染。
2. P1 是 P2、P4 的数据基础；没有不可变 Passage 和 Claim，就无法可靠评分或评测引用。
3. P2 完成后再做 P3，Skill 的质量门禁才能基于可靠性和冲突状态表达。
4. P4 在 P1-P3 形成稳定接口后执行，避免评测对象持续变化。
5. P5 在研究质量链路得到评测证明后推进，避免提前为未验证的能力增加基础设施复杂度。

## 10. 全阶段回归要求

- 每个阶段都必须重跑现有编译检查、单元测试和 smoke 测试。
- 所有 Schema、Policy、Skill、Prompt、Extractor 和评测数据都必须版本化。
- 新能力默认通过特性开关灰度启用，并提供迁移、回滚和兼容方案。
- 任何高置信结论必须可回溯到不可变 Passage、SourceSnapshot 和 ToolTrace。
- 任何无法解决的冲突必须保留为不确定状态，不允许由 LLM 静默裁决。
- P0-P2 保持 SQLite Demo 可运行；P5 再完成 PostgreSQL、对象存储和持久队列迁移。
